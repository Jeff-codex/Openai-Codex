# MASTER OPERATING RULES

이 문서는 `EveryonePR / 모두의피알` 프로젝트를 운영하는 최종 의사결정자용 상위 운영 규칙이다.

## 1. 운영 철학
에이전트는 아이디어를 부풀리는 존재가 아니라, 마스터의 시간과 판단을 절약하는 운영 모듈이다.
따라서 모든 에이전트 출력은 아래 기준을 따라야 한다.

1. 길게 설명하지 말 것
2. 실행 전환 가능한 단위로 말할 것
3. 지금 할 것과 하지 말 것을 분리할 것
4. Codex가 바로 작업할 수 있게 쪼갤 것
5. 프로젝트 본질을 벗어나는 제안은 폐기할 것

## 2. 현재 프로젝트 상태
현재 모두의피알은 아래 기반 공사를 이미 마친 상태로 간주한다.

1. `everyonepr.com` 공개면 전환 완료
2. `review / member / admin` 호스트 전환 완료
3. preview / production `D1 / KV / R2` 분리 완료
4. Cloudflare AI crawl override 완료
5. 공개 SEO / AEO / GEO 구조 확장 완료
6. 핵심 보안 하드닝 완료

따라서 현재 기본 초점은 아래다.

1. 회원전용페이지 UI / UX 정비
2. 주문 전환 및 재주문 흐름 정교화
3. 관리자 운영 효율 개선
4. 검색엔진 후속 추적과 품질 유지

## 3. 보호 영역
아래 영역은 명시적 승인 없이는 확장, 이전, 재설계, 구조 변경을 제안하거나 실행하지 않는다.

- `member`
- `admin`
- `api`
- 결제 / 토스 관련 흐름
- 도메인 / 리디렉션 정책
- Cloudflare `D1 / KV / R2` stateful binding
- production secret / 보안 정책
- 운영 데이터 구조를 흔드는 대규모 리팩터링

## 4. 기본 운영 순서
각 프로젝트는 아래 직렬 순서를 기본으로 한다.

1. GA-1: 지금 성장/권위 병목이 무엇인지 정의
2. DS-1: 신뢰/경험/구조 측면의 문제 해석
3. DO-1: 실제 구현과 운영 단위로 분해
4. RO-1: 수익 또는 사업 구조 연결 검토
5. MO-1: 최종 우선순위와 이번 주 실행안 압축

단, 현재 단계가 회원전용페이지 UX 정비라면 보통 `DS-1 -> DO-1 -> MO-1`이 먼저다.
검색/콘텐츠 병목이 다시 커질 때만 `GA-1`을 우선 호출한다.

## 5. 에이전트 호출 규칙
### 규칙 A
한 번에 모든 에이전트를 동시에 쓰지 않는다.
현재 단계에 맞는 핵심 에이전트 2~3개만 먼저 호출한다.

### 규칙 B
MO-1은 항상 마지막에만 호출한다.
MO-1은 분석 생산자가 아니라 결론 압축기다.

### 규칙 C
DO-1 없이 아이디어만 확장하지 않는다.
Codex로 구현할 수 없는 제안은 우선순위를 낮춘다.

### 규칙 D
이미 끝낸 대공사를 근거 없이 다시 열지 않는다.
예: 공개 도메인 치환, preview/prod binding 분리, AI crawl override, 기본 보안 하드닝.

## 6. 주간 운영 루프
매주 아래 순서로 운영한다.

### 월요일 또는 주초
- `WEEKLY_BRIEF.md` 갱신
- 현재 가장 큰 병목 1개 지정
- 이번 주 목표 3개만 설정

### 실행 중
- 에이전트별 검토 수행
- `BUILD_QUEUE.md` 업데이트
- Codex로 실제 구현 진행

### 주말 또는 주후반
- 이번 주 실행 결과 점검
- `DECISION_LOG.md`에 의사결정 기록
- 다음 주 병목 재정의

## 7. 우선순위 프레임
모든 제안은 아래 4기준으로 본다.

1. Impact — 결과 영향도
2. Feasibility — 지금 구현 가능성
3. Clarity — 얼마나 명확하게 정의되었는가
4. Leverage — 반복 가능성과 확장성

## 8. 버려야 하는 제안의 특징
아래에 해당하면 보류 또는 폐기한다.

- 실제 고객 신뢰와 무관한 기능
- 구현 대비 효과가 낮은 기능
- 초기 단계에서 과도하게 무거운 구조
- 검색 유입은 늘리지만 전환 구조가 없는 전략
- 권위를 해치거나 브랜드를 희석시키는 확장
- 현재 보호 영역을 불필요하게 다시 흔드는 제안

## 9. Codex 브리프 작성 규칙
Codex에 넘길 브리프는 반드시 다음을 포함한다.

- 목적
- 현재 단계
- 구현 범위
- 입력/출력
- 상태 변화
- UI 또는 콘텐츠 구조
- 제외 범위
- 보호 영역
- live 검증 기준
- 완료 기준

## 10. 마스터 최종 판단 규칙
마스터는 항상 다음 순서로 판단한다.

1. 이게 프로젝트 본질에 맞는가
2. 지금 이 단계에서 필요한가
3. 실제로 구현 가능한가
4. 수익/권위/전환에 연결되는가
5. 지금 하지 않는 편이 더 나은가

## 11. 가장 중요한 원칙
- 모두의피알은 `신뢰 -> 주문 -> 재주문` 구조를 먼저 만든다.
- 현재 단계에서는 `회원전용페이지의 신뢰감, 명확성, 주문 용이성`이 핵심이다.
- 에이전트는 완료된 대공사를 무시하고 처음부터 다시 설계하려 들면 안 된다.
